Project Overview

NexusOS is a vertically integrated insurance agent operating system. It is not a generic CRM. It is not GoHighLevel with an insurance skin. It is a purpose-built platform that unifies every workflow a life insurance agent touches — from lead intake to policy bind — with an AI layer that is embedded in the call itself.

CRM Pipeline
GHL-inspired drag-drop stages with auto-triggered SMS/email workflows on stage transitions.
Power Dialer
Browser-based softphone via Twilio Voice SDK. Single-line MVP, parallel dialer deferred for TCPA architecture.
Live Transcription
Deepgram streaming STT with speaker diarization. <2 second latency. Captions render in-call.
AI Underwriting Extraction
Claude tool-use fires every 3 seconds on new transcript chunks. Extracts structured health facts.
Carrier Ranking Engine
Deterministic TypeScript function. Ranks carriers live as health facts emerge. Auditable, E&O-defensible.
Auto-Quote Generator
Post-call quote comparison: top 3 carriers with monthly premiums and the reason each was ranked.
Commission Dashboard
Policy tracking, chargeback alerts, expected vs. paid commission ledger.
Agent Leaderboard
KPIs: calls made, policies written, revenue. Snapshotted per period.
02

The Moat

⚠
The moat is the AI cockpit. This is what makes the product defensible. Generic tools like Cluely, Gong, and Chorus overlay Zoom/Meet with general sales suggestions. We are not doing that.
What We Do That Nobody Else Does
  • Listens for medical conditions during the call and triggers insurance-specific follow-up questions automatically
  • Example: client says "I have diabetes" → AI coaching panel injects: "Type 1 or Type 2? A1C level? When diagnosed? Insulin or metformin?"
  • Maintains a structured underwriting fact panel that updates in real-time as answers come in
  • Re-ranks carriers live as new health facts emerge — different carriers have different underwriting tolerances
  • Post-call: generates a quote comparison showing top 3 carriers with premiums and the reason each was ranked (e.g., "Mutual of Omaha accepts A1C under 7.5 with 3+ years since diagnosis")
  • All of this is deterministic rules, not RAG, not hallucination. Auditable. E&O-defensible.

Why RAG Would Kill Us Here

If the ranking engine hallucinates a carrier that accepts a condition it doesn't actually accept, and an agent relies on that ranking and submits an app, the brokerage has an Errors & Omissions claim. The ranking logic must be deterministic, structured, and auditable. RAG is reserved for Phase 2+ features like objection handling libraries and historical call search.

Data Models

All tables live in Supabase (Postgres). Row-level security is enforced at the database level — agents can only read their own contacts; agency admins can read their agents' data; super-admins see everything.

Core Entities

sql supabase/migrations/001_core.sql 
-- Contacts (leads / clients)
CREATE TABLE contacts (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  first_name          text NOT NULL,
  last_name           text NOT NULL,
  phone               text,
  email               text,
  dob                 date,
  state               char(2),
  created_by_agent_id uuid REFERENCES agents(id),
  created_at          timestamptz DEFAULT now()
);

-- Policies
CREATE TABLE policies (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  contact_id      uuid REFERENCES contacts(id),
  carrier_id      uuid REFERENCES carriers(id),
  product_type    text,  -- term_20, whole_life, etc.
  coverage_amount integer,
  premium         numeric(10,2),
  status          text,  -- active, lapsed, chargedback, pending
  effective_date  date,
  expiry_date     date
);

-- Pipeline
CREATE TABLE pipeline_stages (
  id                uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name              text NOT NULL,
  position          integer,
  auto_sms_template text
);

CREATE TABLE pipeline_cards (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  contact_id uuid REFERENCES contacts(id),
  stage_id   uuid REFERENCES pipeline_stages(id),
  agent_id   uuid REFERENCES agents(id),
  moved_at   timestamptz DEFAULT now()
);

Voice / AI Tables

sql supabase/migrations/002_voice_ai.sql 
-- Call sessions
CREATE TABLE call_sessions (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  agent_id        uuid REFERENCES agents(id),
  contact_id      uuid REFERENCES contacts(id),
  twilio_call_sid text,
  started_at      timestamptz,
  ended_at        timestamptz,
  recording_url   text
);

-- Raw transcripts (per utterance)
CREATE TABLE transcripts (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  call_session_id  uuid REFERENCES call_sessions(id),
  speaker          text,  -- 'agent' | 'client'
  text             text,
  timestamp_ms     integer
);

-- Extracted underwriting facts
CREATE TABLE extracted_facts (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  call_session_id  uuid REFERENCES call_sessions(id),
  contact_id       uuid REFERENCES contacts(id),
  fact_type        text,   -- 'condition', 'tobacco', 'medication', etc.
  fact_key         text,
  fact_value       jsonb,
  confidence       numeric(4,3),
  extracted_at     timestamptz DEFAULT now()
);

-- Live carrier rankings (updated during call)
CREATE TABLE carrier_rankings (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  call_session_id  uuid REFERENCES call_sessions(id),
  carrier_id       uuid REFERENCES carriers(id),
  rank             integer,
  score            numeric(6,3),
  reason           text,
  ranked_at        timestamptz DEFAULT now()
);

-- Post-call quotes
CREATE TABLE quotes (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  call_session_id  uuid REFERENCES call_sessions(id),
  carrier_id       uuid REFERENCES carriers(id),
  product_type     text,
  coverage_amount  integer,
  monthly_premium  numeric(10,2),
  annual_premium   numeric(10,2),
  notes            text
);

AI Tool Schema

The Claude API is called via streaming tool-use on a background interval (every ~3 seconds) as new transcript chunks arrive. The tool fires only when there is new content since the last extraction. The extracted object is merged into the running underwriting facts panel in Supabase Realtime.

json lib/ai/tool-schema.ts → extract_underwriting_facts 
{
  "name": "extract_underwriting_facts",
  "description": "Extract medical and demographic facts from an insurance call transcript for underwriting evaluation",
  "input_schema": {
    "type": "object",
    "properties": {
      "age":             { "type": "integer" },
      "gender":          { "type": "string", "enum": ["male", "female"] },
      "state":           { "type": "string", "description": "2-letter US state code" },
      "coverage_amount": { "type": "integer", "description": "Requested coverage in dollars" },
      "coverage_type": {
        "type": "string",
        "enum": ["term_10", "term_20", "term_30", "whole_life", "iul", "ul", "final_expense"]
      },
      "conditions": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "condition": { "type": "string" },
            "details":   { "type": "object" }
          }
        }
      },
      "tobacco_status":   { "type": "string", "enum": ["current", "former", "never"] },
      "tobacco_quit_date": { "type": "string", "format": "date" },
      "medications":      { "type": "array", "items": { "type": "string" } },

MVP Demo Scope

The MVP is a live demo, not a production system. It must perform a scripted end-to-end walkthrough flawlessly. Every feature listed below must work on demo day with seeded data.

The Demo Script (8 Steps)
  1. Agent opens cockpit → sees lead in pipeline board
  2. Clicks "Dial" → browser softphone connects to client
  3. Client mentions age + coverage amount → carrier shortlist populates
  4. Client says "I have Type 2 diabetes" → coaching panel injects follow-up questions
  5. Client answers A1C, meds, diagnosis date → fact panel updates, carriers re-rank in real-time
  6. Client mentions smoking history → another re-rank with animated transition
  7. Call ends → quote comparison renders: top 3 carriers with monthly premiums + ranked reasons
  8. Agent drags pipeline card → SMS fires to client's phone

Compliance Requirements

⚠
Do not ship any dialer feature without legal review. TCPA violations carry $500–$1,500 per call in statutory damages. At scale this is existential.
TCPA TSR HIPAA-adjacent (GLBA/NAIC) E&O Carrier contracts ACORD State insurance regs
TCPA
Provable per-number consent required. DNC scrubbing (federal + state) before any dial. Recording disclosures per state. Two-party consent states: CA, FL, IL, MD, MA, MT, NH, PA, WA.
Medical Data (HIPAA-adjacent)
We capture A1C, diagnoses, prescriptions. Not technically HIPAA (insurance = GLBA/NAIC), but treat as PHI. Column-level encryption, audit logs, retention policies required.
E&O — Carrier Ranking
If ranking is wrong and agent relies on it, the brokerage has Errors & Omissions exposure. All ranking logic must be deterministic and auditable. No RAG, no LLM in the ranking path.
State Insurance Regulations
AI "recommending" a carrier must be framed as agent-selected from AI-ranked options. The licensed agent is the actor. The AI is a tool. This distinction must be in the UI copy.
Carrier Contracts
Redistribution of carrier feed data may be restricted by contract. Get legal signoff before connecting carrier SFTPs in Phase 2.
ACORD Forms
ACORD forms are licensed IP. We cannot generate Certificates of Insurance without rights. Deferred entirely.

RAG / Knowledge Architecture

Real-time Ranking — NO RAG
Deterministic TypeScript function querying structured Postgres tables. Sub-100ms. Auditable. No vector search in this path, ever.
Phase 2 — Sales RAG
pgvector in Supabase for: sales scripts library, objection handling, historical call transcript search ("similar diabetic cases that closed").
Phase 3 — Carrier Guide Ingestion
RAG pipeline over 300-page carrier underwriting PDFs → human-reviewed extraction → structured rules tables. RAG feeds the DB; never serves agents directly.
→
No Neo4j. Postgres with foreign keys is the knowledge graph for this product. pgvector handles semantic search when we need it.

Key Decisions Already Made

These decisions are finalized. Do not re-litigate them during the MVP build. Raise concerns in a retrospective after the demo.

  • 1
    Deterministic rules for underwriting — NOT RAG
    Auditable, fast (<100ms), E&O-defensible. RAG hallucination in a ranking context creates legal liability.
  • 2
    Supabase over custom backend
    RLS, Realtime, pgvector, Edge Functions, Auth all in one platform. Fewer moving parts for a 3-week build.
  • 3
    Browser softphone (Twilio Voice SDK) over hardware integration
    Simpler architecture, demo-able without physical setup, works in any browser. Hardware integration is Phase 2+.
  • 4
    Claude tool-use for extraction — not a fine-tuned model
    Flexibility to iterate on the schema without retraining. No training pipeline to maintain. Tool-use is production-grade.
  • 5
    Dark theme, insurance-professional aesthetic
    The buyer is an insurance brokerage, not a startup. Design signals trust and authority, not playfulness.
  • 6
    Internal tool first, product later
    Dramatically reduces compliance surface during MVP. No external user registration, no multi-tenant billing, no consumer data exposure.
  • 7
    Single-line dialer first
    Parallel dialing requires robust TCPA consent architecture and DNC infrastructure. Deferred to prevent legal exposure on day one.
